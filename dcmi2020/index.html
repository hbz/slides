Why an LRMI profile?

Great interest in LRMI in the German-speaking OER community but there are no easy instructions for implementation

Current project by hbz and TIB Hannover in need of a standards-oriented JSON Schema for learning resources: OERSI – OER Search Index

The general approach

Machines first, humans second

1st step: define JSON Schema & controlled vocabularies, and provide SkoHub-based reference web form

2nd step: write human-readable documentation with ReSpec

JSON Schema plus SKOS vocabularies

JSON Schema defines fields, value type & referenced controlled vocabularies

Controlled vocabularies/value lists are published separately as SKOS files and are referenced in the schema

Schema can be tried out in SkoHub editor including lookup of controlled vocabs

JSON Schema & SkoHub Editor

Property with non-specified string


{
  "properties": {
    "name": {
    "title": "Title",
    "type": "string"
    }
  }
}

Property with formated string (URI)


{
  "image": {
    "title": "Image",
    "type": "string",
    "format": "uri",
    "_display": {
	  "placeholder": "A link to an image of the resource"
    }
  }
}

Listing possible strings


{
  "type": {
    "title": "Type",
    "type": "string",
    "enum": [
      "AudioObject",
      "Book",
      "Course",
      "CreativeWork",
      "DataDownload",
      "ImageObject",
      "PresentationDigitalDocument",
      "SoftwareApplication",
      "VideoObject"
    ]
  }
}

Referencing a SKOS vocabulary


{
  "about":{
    "title":"Subject",
    "type":"array",
    "items":{
      "type":"object",
      "properties":{
        "inScheme":{
          "type":"object",
          "properties":{
            "id":{
              "type":"string",
              "enum":[
                "https://w3id.org/kim/hochschulfaechersystematik/scheme"
              ]
            }
          }
        }
      },
      "_widget":"SkohubLookup"
    }
  }
}

Automatic test setup

Profile is automatically tested against two sets of examples (valid & invald)

Whenever a file validates that shouzldn't it is added to the /invalid folder and the schema is updated to pass the tests

The examples are also helpful for people implementing the profile

Where exactly do these suggested values come from?

SKOS & SkoHub Vocabs

Publishing SKOS with SkoHub vocabs

Controlled vocabularies must be published with SkoHub Vocabs

Publication & maintenance in Turtle file on GitHub/GitLab & publication as HTML/JSON-LD under skohub.io (siehe here for details)

Pros and cons

Benefits

Development on GitHub/GitLab with established working processes

Schema and reference web form are developed in one go

Controlled vocabularies are developed parallely in SKOS and can be used independently from the Schema

Possible drawbacks

Schema focuses on one specific JSON-LD representation of the data

Thus, no way to validate differently structured JSON or other RDF serializations

Human-readable version is not there from the beginning & must be written separately
